# Proposal: Nightly monitoring matrix

## Context

This proposal adds a **nightly monitoring job** to cube-registry CI. It complements
the existing PR-level slow-check (which runs one cube's debug episode after merge)
by running *all* registered cubes against *all* their supported infra targets on a
nightly schedule, and publishing results to a persistent journal and dashboard.

The PR-level smoke tests for cubes in cube-harness's own `cubes/` directory are a
separate concern; they are specified in
[cube-harness openspec/changes/cube-ci/proposal.md](https://github.com/The-AI-Alliance/cube-harness/blob/main/openspec/changes/cube-ci/proposal.md).
This document covers the registry-level nightly job only.

---

## What

A nightly GitHub Actions workflow (`nightly-monitoring.yml`) runs `cube test <name>`
for every entry in the registry, across each `supported_infra` target declared by
that entry's YAML, and appends structured results to a persistent journal.

A dashboard page (extending the existing static site) renders the matrix as a
pass/fail table with per-cell drill-down to full run logs.

---

## Matrix: cube × infra

Rows = registered cubes, columns = infra targets. A cell is only exercised if the
cube's `supported_infra` list includes that target. Sparse matrix is expected and
correct — not every cube makes sense on every infra.

Representative cells (not exhaustive):

```
              | gh-runner | docker | aws | azure | hf-hosted |
miniwob       |    ✓      |        |     |       |           |
arithmetic    |    ✓      |        |     |       |           |
terminalbench |    ✓      |   ✓    |     |       |           |
workarena     |           |        |     |       |     ✓     |
swebench-*    |    ✓      |   ✓    |     |       |           |
osworld       |           |        |  ✓  |   ✓   |           |
webarena-*    |           |        |  ✓  |   ✓   |           |
```

---

## Statistics collected per cell

For each (cube, infra) pair that is exercised:

- **Pass / fail** — with auto-retry (default 2×, configurable per cube)
- **L1 provisioning time** — dependency install
- **L2 provisioning time** — benchmark setup (e.g. dataset download, server start)
- **L3 provisioning time** — per-task reset latency
- **Episode wall-clock time** — from first step to final reward

The retry result is recorded (e.g. `pass_on_retry_2` vs `fail_all`), enabling
flakiness detection across runs.

---

## Journal

Every nightly run appends one record per (cube, infra, version) cell to a structured
log (`nightly-results/<cube-id>/<infra>/<date>.json`). This enables:

- Trend analysis (how often does a cube fail over time?)
- Flakiness detection (pass on retry 1, fail on retry 2?)
- Provisioning time regressions (did L2 time spike after a cube version bump?)
- Cross-infra comparison for cubes that support multiple targets

Journal files are written by the CI bot (same path-restriction as `stress-results/`).

---

## Dashboard

An additional page in the static site renders the nightly matrix:

- Rows = cube name + current version, columns = infra targets
- Cell color: green (all pass), yellow (pass on retry), red (fail all), grey (not applicable)
- Cell is clickable → shows the full log for that (cube, infra, date) run
- "Latest run" status is also surfaced on each cube's individual registry page

The dashboard page is generated by `site-src/generate.py` from the journal files and
committed by `generate-site.yml` alongside the entry updates.

---

## What runs nightly

1. **Published version**: All cubes in the registry pinned to the version in their
   `entries/<id>.yaml`.
2. **Dev version** (optional, lower priority): Cubes whose `cubes/<id>/` directory in
   cube-harness differs from their published PyPI version. This is gated on a
   version comparison check before running, so the job is a no-op for stable cubes.

Both modes run `cube test <name>` (the debug agent), never an LLM-based agent. No LLM
API keys are required.

---

## Required secrets

Common infrastructure secrets (AWS, Azure, HF-hosted) are pre-configured in
cube-registry's GitHub secret store. Cube-specific secrets are registered alongside
the cube submission (new field: `ci.required_secrets`, see Cube-standard delta below).

The nightly job reads `ci.required_secrets` for each cube and skips cells where the
required secret is not configured, rather than failing.

---

## Cube metadata needed (requires cube-standard change)

To make the matrix declarative rather than hardcoded, `BenchmarkMetadata` gains a
`ci` block:

```python
class CIConfig(TypedBaseModel):
    tier: int                          # 1=every-PR  2=nightly  3=weekly
    supported_infra: list[str] = []    # e.g. ["gh-runner", "docker", "aws"]
    required_secrets: list[str] = []   # env var names for non-standard secrets
    service_containers: list[str] = [] # docker-compose services needed for setup
```

`cube test --list --tier=1` emits the tier-1 cube list. This is a cube-standard
change; a corresponding openspec proposal will be filed there.

Until cube-standard lands `CIConfig`, the nightly job uses a hardcoded matrix derived
from the `supported_infra` field already present in `entries/<id>.yaml`.

---

## Security

The nightly job follows the same security model as slow-check:
- Runner has cloud credentials but **never imports the benchmark package** in the
  runner process.
- Package runs inside the provisioned VM / container.
- Journal write access is restricted to the CI bot.
